# Writing Effective Tools for Agents：Agent 工具设计原则

资料来源：[Anthropic Engineering: Writing effective tools for agents — with agents](https://www.anthropic.com/engineering/writing-tools-for-agents)

## 阅读目标

关注三个问题：

1. 为什么面向 agent 的工具不能简单等同于后端 API 端点。
2. 如何通过评测和 agent 轨迹分析改进工具设计。
3. 工具命名、描述、参数和返回值如何影响模型选择工具、调用工具和恢复错误。

核心结论是：工具是确定性系统与非确定性 agent 之间的接口。好的工具不只是“可调用”，还要让模型容易理解用途、选择正确动作、获取足够上下文，并在失败时知道下一步如何恢复。

## 名词解释

| 名词 | 解释 | 简单例子 |
|---|---|---|
| Tool | 暴露给 agent 的可调用能力，通常由应用代码执行真实查询、写入或外部系统操作。 | `search_customer_orders` 查询用户订单。 |
| Agent | 能根据目标、上下文和工具结果循环决策的软件组件。 | 客服 agent 先查订单，再判断是否需要退款。 |
| API endpoint | 后端系统面向程序调用的接口，通常按资源和操作拆分。 | `GET /orders/{id}`、`POST /refunds`。 |
| Tool definition | 提供给模型的工具名称、说明、参数 schema 和返回结构。 | `name`、`description`、`input_schema`。 |
| Tool namespace | 通过命名前缀或分组表达工具所属服务、资源或能力域。 | `asana_search_tasks`、`jira_create_issue`。 |
| Eval | 用任务集和评分规则评估 agent 是否能完成目标。 | 给 50 个真实客服任务，统计工具选择和最终答案正确率。 |
| Agent transcript | agent 执行过程中的完整轨迹，包括模型思考、工具调用、工具返回和最终输出。 | 一次订单排障的 tool call 序列和 observation。 |
| Token-efficient response | 控制工具返回长度，只保留下一步决策需要的信息。 | 搜索工具只返回标题、ID、状态和更新时间，而不是完整正文。 |

## 1. 背景：工具是 agent 的上下文边界

很多系统会把已有 API endpoint 直接包装成 tool。这样做原型成本低，但容易把后端系统的资源拆分、字段命名和分页细节暴露给模型。

对确定性代码来说，多个细粒度 API endpoint 可以通过程序逻辑稳定编排；对 agent 来说，每多一个工具、参数或返回字段，都会增加选择成本和上下文噪声。工具设计的重点不是复刻后端 API，而是为 agent 提供适合当前任务的动作边界和信息密度。

原文提出的工作流可以概括为：

| 步骤 | 目的 | 产出 |
|---|---|---|
| 构建原型 | 快速暴露一组可用工具。 | 初始工具定义和工具执行器。 |
| 运行评测 | 用真实任务检查 agent 是否能完成目标。 | 成功率、失败案例、错误分布。 |
| 分析轨迹 | 从 transcript 中定位工具使用障碍。 | 命名误解、参数误填、返回值噪声、缺失上下文。 |
| 迭代工具 | 修改工具定义、参数、返回结构或工具粒度。 | 更容易被 agent 正确使用的工具集。 |

这里的关键不是一次性设计出完美工具，而是建立“工具设计 -> agent 执行 -> 轨迹分析 -> 工具修正”的闭环。

## 2. 用评测驱动工具设计

工具是否好用，不能只看接口是否能跑通，而要看 agent 在完整任务中是否能稳定完成目标。

原文建议把评测设计成贴近真实任务的端到端场景。例如，不只测试 `search_tasks` 是否返回结果，而是测试 agent 能否根据用户要求找到正确任务、判断状态、更新字段并给出可解释的回复。

| 评测维度 | 检查项 | 期望状态 |
|---|---|---|
| 任务真实性 | 任务是否来自真实用户请求或业务流程。 | 能暴露真实语言歧义、数据缺失和多步骤依赖。 |
| 结果可验证 | 是否有明确的正确结果或人工可判定标准。 | 能判断 agent 最终是否完成任务，而不是只看是否调用工具。 |
| 轨迹可分析 | 是否记录完整 tool call、参数和返回值。 | 失败时能定位是工具选择错、参数错、信息不够还是返回噪声。 |
| 覆盖面 | 是否覆盖常见路径、边界条件和失败路径。 | 工具改动不会只优化单个 happy path。 |

评测之后，重点看 agent 的行为轨迹：

- agent 是否在多个相似工具之间犹豫或误选。
- 参数名是否诱导 agent 填错。
- 工具返回是否让 agent 找不到关键 ID、状态或下一步动作。
- 错误信息是否只说明失败，而没有告诉 agent 如何修正。
- agent 是否因为返回内容过长而忽略真正重要的字段。

这些信号比单纯观察最终答案更有价值。最终答案只能说明“这次成败”，轨迹能说明“为什么成败”。

原文也强调可以让编码 agent 参与这个循环：把失败用例、transcript、当前工具定义和期望结果交给 agent，让它提出工具描述、参数或返回结构的修改建议。这里仍然需要工程师审查，因为 agent 适合发现模式和生成候选修改，不应替代对权限、业务语义和兼容性的判断。

## 3. 原则一：选择合适的工具粒度

面向 agent 的工具应该围绕任务工作流设计，而不是围绕后端资源表设计。

如果一个任务总是需要先搜索客户、再查订单、再查物流，直接暴露三个底层工具会把编排责任交给模型。更适合的方式可能是提供一个 `lookup_order_context` 工具，由确定性代码完成多源查询，并返回 agent 判断下一步需要的高密度上下文。

| 设计取向 | 优点 | 风险 |
|---|---|---|
| 细粒度工具 | 接近底层 API，复用性高。 | 工具数量多，agent 需要自己编排，容易漏步骤或选错工具。 |
| 工作流工具 | 贴近用户任务，返回信息集中。 | 需要额外适配层，过度封装后可能降低灵活性。 |
| 分层工具 | 常见任务用工作流工具，少数高级任务保留底层工具。 | 需要清晰命名和适用边界，否则仍会混淆。 |

工程上可以先问三个问题：

1. 这个工具是否对应 agent 经常需要完成的一类任务。
2. agent 调用它之后，是否能获得足够信息做下一步判断。
3. 如果把几个底层 API 合并，是否能显著减少模型的选择和编排负担。

工具数量不是越多越强。工具集越大，模型越容易把相似能力混淆，也更容易把无关工具纳入推理路径。

## 4. 原则二：用命名和命名空间减少歧义

工具名是模型选择工具时最先看到的信号。命名不清晰时，模型可能在正确工具和相似工具之间误选。

好的命名通常包含三类信息：

| 命名信息 | 作用 | 示例 |
|---|---|---|
| 所属系统 | 区分能力来源。 | `jira_`、`slack_`、`github_`。 |
| 操作意图 | 说明工具做什么。 | `search`、`create`、`update`、`archive`。 |
| 业务对象 | 说明工具作用于什么资源。 | `issue`、`message`、`pull_request`。 |

例如 `github_search_pull_requests` 比 `search` 更容易让模型理解工具边界。对于跨系统 agent，命名空间尤其重要，因为不同系统可能都有 search、create、update 等通用动作。

命名也需要通过 eval 验证。对某些模型和任务，前缀比后缀更容易被识别；对另一些任务，资源名靠前可能更清晰。命名不是纯审美问题，而是模型可用性问题。

## 5. 原则三：返回有用上下文，而不是原始数据倾倒

工具返回值会直接进入 agent 的上下文窗口。返回越全，不一定越有帮助。

原始 API response 往往包含大量字段：内部 ID、审计字段、嵌套对象、空值、调试信息。人类工程师可以筛选这些字段，模型却可能把噪声当成线索，或者被长返回淹没。

工具返回应优先服务下一步决策：

| 返回内容 | 适合保留的原因 | 示例 |
|---|---|---|
| 稳定标识符 | 支持后续工具调用。 | `order_id`、`task_id`。 |
| 人类可读名称 | 支持模型解释和用户确认。 | 订单标题、任务名称、客户名。 |
| 当前状态 | 支持分支判断。 | `paid`、`shipped`、`blocked`。 |
| 时间和来源 | 支持判断新旧、引用和审计。 | `updated_at`、`source_url`。 |
| 下一步提示 | 支持错误恢复或继续操作。 | “如需退款，请调用 `create_refund_request`”。 |

当返回可能很长时，应提供过滤、分页、摘要或格式参数。例如：

```json
{
  "query": "late shipment",
  "limit": 5,
  "response_format": "summary"
}
```

这里的目标不是隐藏信息，而是让工具默认返回高信号内容，并允许 agent 在确实需要时请求更多细节。

## 6. 原则四：把工具说明当作 prompt 工程

工具描述不是普通 API 文档。它会被模型读取，并影响模型是否调用、何时调用和如何填参。

有效的工具说明通常要写清楚：

| 维度 | 检查项 | 期望状态 |
|---|---|---|
| Purpose | 是否说明工具解决的任务。 | 模型能区分它和相似工具。 |
| Use When | 是否说明适用场景。 | 模型知道什么时候应该调用。 |
| Do Not Use When | 是否说明禁用边界。 | 模型能避开高风险或无关场景。 |
| Parameters | 参数名、类型、默认值和约束是否清楚。 | 模型能生成可校验输入。 |
| Returns | 返回字段是否稳定且可解释。 | observation 能支持下一步决策。 |
| Errors | 错误是否给出可恢复动作。 | agent 知道重试、改参、换工具或停止。 |

参数命名也属于 prompt 工程。`user_query`、`customer_email`、`max_results` 这类名称比 `q`、`id`、`n` 更容易让模型正确填充。描述中应避免只写后端实现细节，而要说明参数在任务语义中的作用。

错误信息同样重要。一个只返回 `400 Bad Request` 的工具会让 agent 难以恢复；更好的返回是说明哪个字段无效、允许的取值是什么、是否可以重试。

## 7. 与 Tool Card 的关系

本仓库已有 [Tool Card 模板](../react-framework/tool-card-template.md)，更偏“单个工具说明应该如何写”。Anthropic 这篇文章补充的是“工具集应该如何被 agent 使用、评测和迭代”。

两者可以合在一起使用：

| 层次 | 对应文档 | 解决的问题 |
|---|---|---|
| 单工具说明 | Tool Card 模板 | 一个工具的用途、边界、schema、错误和安全要求如何写清楚。 |
| 工具集设计 | 本文档 | 多个工具如何选粒度、命名、返回上下文，并通过 eval 持续优化。 |
| Agent 系统设计 | 12-Factor Agents | 工具调用如何进入状态、控制流、暂停恢复和人工介入。 |

也就是说，Tool Card 是工具的接口说明；Writing Effective Tools for Agents 更像工具系统的设计和评测方法。

## 8. 工程落地检查表

| 维度 | 检查项 | 期望状态 |
|---|---|---|
| 工具粒度 | 是否把常见多 API 编排封装成工作流工具。 | agent 不需要反复猜测底层调用顺序。 |
| 工具数量 | 是否移除重复、低频或只给人类调试用的工具。 | 默认工具集短而清晰。 |
| 命名空间 | 跨系统工具是否带服务或资源前缀。 | 模型能区分相似动作。 |
| 参数设计 | 参数名是否语义化，枚举和范围是否明确。 | 输入可被模型生成，也可被程序校验。 |
| 返回结构 | 是否默认返回高信号字段和后续动作线索。 | observation 支持下一步决策。 |
| 错误恢复 | 错误是否告诉 agent 如何修正。 | 失败后能重试、改参、换工具或停止。 |
| 评测闭环 | 是否有真实任务集和 transcript 分析流程。 | 工具改动能用数据验证，而不是凭感觉。 |

## 关键结论

1. 面向 agent 的 tool 不是 API endpoint 的简单包装，而是为模型决策设计的上下文接口。
2. 工具设计要通过真实任务 eval 验证，重点分析 agent transcript 中的误选、误填、噪声和恢复失败。
3. 工具粒度应贴近任务工作流，常见多步查询可以由确定性代码封装，减少模型编排负担。
4. 工具命名、命名空间、参数名和描述都会影响模型行为，应当像 prompt 一样迭代。
5. 工具返回值应默认高信号、可追溯、可继续操作，避免把原始 API response 直接倒入上下文。
